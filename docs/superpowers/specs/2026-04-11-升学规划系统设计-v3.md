## 1. 项目概述

### 1.1 愿景

> 基于国家战略发展方向，整合教育政策与高校录取数据，为初中/高中学生提供个性化升学规划方案的本地运行工具。

让每个家庭都能提前 5-10 年规划升学路径，不是"考多少分报什么学校"，而是"想成为什么样的人，选择什么路径"。

### 1.2 目标用户

| 用户群体 | 特征 | 核心诉求 |
|---------|------|---------|
| **初中生家长**（主要） | 孩子 12-15 岁，教育意识强 | 了解未来趋势、制定培养路径、提前准备多元升学 |
| **升学顾问/教师**（次要） | 学校老师、机构顾问 | 批量分析学生、生成规划报告、跟踪政策变化 |
| **教育机构**（商业） | 培训机构、咨询公司 | 专业化规划工具、提升服务质量 |

### 1.3 核心价值

```
对家长：
✓ 提前 5-10 年规划，避免临时抱佛脚
✓ 识别国家发展趋势，选择正确方向
✓ 多元升学途径，不局限于高考裸分
✓ 三档目标 + 行动方案，清晰可执行

对机构：
✓ 专业化规划工具，提升服务质量
✓ 数据驱动决策，增强说服力
✓ 可复制的规划流程，提高效率
✓ 本地部署，数据安全
```

### 1.4 产品定位

| 维度 | 定位 |
|------|------|
| **市场定位** | MVP 阶段：小范围分享 → 产品化：教育机构合作 |
| **技术定位** | MVP: Python + Streamlit + DuckDB（快速验证）<br>产品化：FastAPI + React + PostgreSQL（商业化） |
| **数据定位** | 全国 985/211 高校（约 120 所）在辽宁录取数据 |
| **时间投入** | 3-6 个月，副业开发 |
| **质量标准** | 可商业化交付 |

### 2.3 核心引擎架构

```python
# 规划引擎核心流程
规划引擎工作流程：
├── 1. 路线穷举器 (RouteEnumerator)
│   ├── 输入：学生画像 + 政策库
│   ├── 处理：生成 50-100 条可能路线
│   └── 输出：路线列表
│
├── 2. 匹配引擎 (MatchingEngine)
│   ├── 兴趣匹配 (40%)
│   ├── 能力匹配 (25%)
│   ├── 经济匹配 (20%)
│   ├── 时间匹配 (10%)
│   └── 地域匹配 (5%)
│
├── 3. 目标生成器 (TargetGenerator)
│   ├── 高目标（冲刺，录取概率 30-50%）
│   ├── 中目标（稳妥，录取概率 50-80%）
│   └── 低目标（保底，录取概率 80-95%）
│
└── 4. 可行性评估 (FeasibilityAssessor)
    ├── 录取概率计算
    ├── 风险评估
    └── 备选方案生成
```

## 4. 数据设计

### 4.1 高校数据表 (universities)

```sql
CREATE TABLE universities (
    -- 基本信息
    id INTEGER PRIMARY KEY,
    name VARCHAR(100) NOT NULL,           -- 高校名称
    code VARCHAR(10) UNIQUE,              -- 教育部代码
    location VARCHAR(50),                 -- 所在地
    type VARCHAR(20),                     -- 类型：综合/理工/师范等
    
    -- 标签
    is_985 BOOLEAN DEFAULT FALSE,         -- 985 高校
    is_211 BOOLEAN DEFAULT FALSE,         -- 211 高校
    is_double_first_class BOOLEAN DEFAULT FALSE,  -- 双一流
    project_type VARCHAR(20),             -- 双一流类型：A 类/B 类
    
    -- 其他信息
    website VARCHAR(200),                 -- 官网
    description TEXT,                     -- 简介
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_univ_type ON universities(is_985, is_211);
CREATE INDEX idx_univ_location ON universities(location);
```

**示例数据**:
```
id | name | code | location | type | is_985 | is_211 | is_double_first_class
1  | 清华大学 | 10003 | 北京 | 综合 | TRUE | TRUE | TRUE
2  | 北京大学 | 10001 | 北京 | 综合 | TRUE | TRUE | TRUE
3  | 东北大学 | 10145 | 辽宁沈阳 | 理工 | TRUE | TRUE | TRUE
```

### 4.2 专业数据表 (majors)

```sql
CREATE TABLE majors (
    id INTEGER PRIMARY KEY,
    university_id INTEGER REFERENCES universities(id),
    name VARCHAR(100) NOT NULL,           -- 专业名称
    code VARCHAR(10),                     -- 专业代码
    category VARCHAR(50),                 -- 专业类别（计算机类、机械类等）
    degree VARCHAR(20),                   -- 学位门类（工学、理学等）
    duration INTEGER DEFAULT 4,           -- 学制（年）
    
    -- 学科评估
    discipline_rank VARCHAR(5),           -- 学科评估等级（A+、A、A-等）
    is_national_key BOOLEAN DEFAULT FALSE, -- 国家重点学科
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_major_univ ON majors(university_id);
CREATE INDEX idx_major_category ON majors(category);
```

### 4.3 录取分数线表 (admission_scores)

```sql
CREATE TABLE admission_scores (
    id INTEGER PRIMARY KEY,
    university_id INTEGER REFERENCES universities(id),
    major_id INTEGER REFERENCES majors(id),
    
    -- 年份和省份
    year INTEGER NOT NULL,                -- 年份（2021-2025）
    province VARCHAR(50) NOT NULL,        -- 省份（辽宁）
    
    -- 科类
    category VARCHAR(20) NOT NULL,        -- 物理类/历史类
    
    -- 分数数据
    min_score INTEGER,                    -- 最低录取分
    max_score INTEGER,                    -- 最高录取分
    avg_score INTEGER,                    -- 平均录取分
    min_rank INTEGER,                     -- 最低位次（关键指标）
    max_rank INTEGER,                     -- 最高位次
    plan_count INTEGER,                   -- 计划招生人数
    actual_count INTEGER,                 -- 实际录取人数
    
    -- 批次
    batch VARCHAR(20),                    -- 批次（本科批/一段等）
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(university_id, major_id, year, province, category)
);

CREATE INDEX idx_score_year ON admission_scores(year);
CREATE INDEX idx_score_univ ON admission_scores(university_id);
CREATE INDEX idx_score_rank ON admission_scores(min_rank);
```

**示例数据**:
```
university_id | major_id | year | province | category | min_score | min_rank | plan_count
1 (清华)      | 1 (计算机) | 2025 | 辽宁 | 物理类 | 695 | 50 | 2
1 (清华)      | 2 (经管) | 2025 | 辽宁 | 物理类 | 690 | 80 | 3
```

### 4.4 学生档案表 (students)

```sql
CREATE TABLE students (
    id INTEGER PRIMARY KEY,
    name VARCHAR(50) NOT NULL,            -- 学生姓名（可化名）
    student_code VARCHAR(20) UNIQUE,      -- 学号（自定义）
    
    -- 基本信息
    gender VARCHAR(10),                   -- 性别
    birth_date DATE,                      -- 出生日期
    grade VARCHAR(20),                    -- 年级（初一/初二/初三/高一/等）
    school VARCHAR(100),                  -- 学校
    city VARCHAR(50),                     -- 城市（沈阳/大连等）
    
    -- 成绩信息
    category VARCHAR(20),                 -- 选科：物理类/历史类
    total_score DECIMAL(5,2),             -- 总分
    ranking INTEGER,                      -- 年级/市排名
    chinese_score DECIMAL(5,2),           -- 语文
    math_score DECIMAL(5,2),              -- 数学
    english_score DECIMAL(5,2),           -- 英语
    other_scores JSON,                    -- 其他科目成绩 {物理：90, 化学：85}
    
    -- 兴趣特长
    interests JSON,                       -- 兴趣列表 ["编程", "机器人", "音乐"]
    specialities JSON,                    -- 特长列表 ["信息学竞赛省二", "钢琴十级"]
    awards JSON,                          -- 奖项列表
    
    -- 家庭情况
    family_budget DECIMAL(10,2),          -- 教育预算（年）
    preferred_locations JSON,             -- 偏好城市 ["北京", "上海"]
    constraints JSON,                     -- 限制条件 {经济：中等，时间：充足}
    
    -- 规划状态
    planning_status VARCHAR(20),          -- 规划状态：未开始/进行中/已完成
    last_plan_date TIMESTAMP,             -- 最后规划时间
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 4.5 规划路线表 (planning_routes)

```sql
CREATE TABLE planning_routes (
    id INTEGER PRIMARY KEY,
    student_id INTEGER REFERENCES students(id),
    
    -- 路线信息
    route_name VARCHAR(100),              -- 路线名称
    route_type VARCHAR(50),               -- 类型：普通高考/强基计划/科技特长生/等
    
    -- 路线详情
    stages JSON,                          -- 阶段规划 [{阶段：初中，目标：...}, {阶段：高中，目标：...}]
    requirements JSON,                    -- 要求条件 {成绩要求，竞赛要求，等}
    timeline JSON,                        -- 时间安排
    
    -- 匹配结果
    match_score DECIMAL(5,2),             -- 匹配度（0-100）
    match_details JSON,                   -- 匹配详情 {兴趣：40, 能力：25, ...}
    
    -- 目标高校
    target_universities JSON,             -- 三档目标 {高：[...], 中：[...], 低：[...]}
    
    -- 可行性
    feasibility_score DECIMAL(5,2),       -- 可行性评分
    success_probability DECIMAL(5,2),     -- 成功概率（%）
    risk_level VARCHAR(20),               -- 风险等级：低/中/高
    
    -- 方案状态
    is_primary BOOLEAN DEFAULT FALSE,     -- 是否主推荐方案
    status VARCHAR(20),                   -- 状态：草稿/已确认/已放弃
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 9. 风险管理

### 9.1 风险清单

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **分数线数据获取困难** | 中 | 高 | 多渠道收集、手工录入备选、优先最新数据 |
| **算法效果不达预期** | 中 | 中 | 先规则引擎后优化、保留人工调整接口 |
| **政策理解偏差** | 中 | 中 | 标注数据来源、人工复核、专家咨询 |
| **开发进度延误** | 低 | 中 | 明确 MVP 范围、可裁剪功能、优先可用性 |
| **Streamlit 性能瓶颈** | 低 | 低 | 数据缓存、懒加载、必要时升级架构 |

### 9.2 应对策略

```
数据风险应对:
1. 准备 3+ 数据源（阳光高考、辽宁招考、高校官网）
2. 开发手工录入界面作为备选
3. 优先保证 2024-2025 最新数据
4. 历史数据可逐步补充

技术风险应对:
1. 使用成熟技术栈（Streamlit+DuckDB）
2. 核心算法先实现基础版本
3. 保留架构升级空间（MVP→产品化）
4. 关键模块编写单元测试

进度风险应对:
1. 明确 MVP 范围（P0 功能）
2. P1/P2 功能可延后
3. 每两周检查进度
4. 必要时缩减数据范围（先做 50 所高校）
```

### 9.3 技术债务管理

```
允许的技术债务（MVP 阶段）:
✓ 硬编码配置（后续移到配置文件）
✓ 简化的算法逻辑（后续优化）
✓ 基础 UI 样式（后续美化）
✓ 最少的错误处理（后续完善）

不允许的技术债务:
✗ 数据库设计缺陷（后期难改）
✗ 核心算法逻辑错误
✗ 数据准确性问题
✗ 安全隐患

债务偿还计划:
- Phase 4 预留 1 周重构时间
- 每个 Phase 结束前 review 技术债务
- 产品化前必须偿还所有债务
```
